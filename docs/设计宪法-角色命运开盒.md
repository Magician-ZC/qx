# 《群像》设计宪法 —— 角色命运开盒

> **这是项目的最高设计文件，凌驾于一切细节设计之上。** 任何系统、玩法、商业、引擎决策若与本宪法冲突，以本宪法为准。
>
> 其余文档据此重新定位：[`大世界沙盘设计方案.md`](大世界沙盘设计方案.md) 是「**后台发动机**」的设计（不是前台玩法）；[`游戏开发方案GDD.md`](游戏开发方案GDD.md) / [`产品方案PRD.md`](产品方案PRD.md) / [`验证实验设计.md`](验证实验设计.md) 是其展开。

## 0. 定调（一句话）

**把《群像》从「大世界沙盘」改成「角色命运开盒游戏」。**

完整世界是后台发动机；玩家真正玩的，是**一个人如何在这个世界里慢慢变成另一个人**。

- 完整世界（无限 agent / 势力 / 战争 / 贸易）**在背后运行**，唯一事实源仍是 `unit.Status` + `events`。
- 但**前台永远从一个角色的一段命运开始**，大世界**不能直接扑到玩家脸上**。
- 世界为角色命运服务，不是反过来：每个大系统都要过一道翻译——「**它会不会制造玩家和角色之间的关系变化？**」战争本身不进前台，「**她朋友被征召了**」才进。

## 1. 十条支柱（不可违背）

> 来自项目负责人，逐条为约束。每条下方是**操作化落点**（§3–§7 详述）。

1. **大世界不能直接扑到玩家脸上。** 玩家第一天只需知道：这是我的人 / 她想要什么 / 她怕什么 / 她今天做了一件我没想到但合理的事 / 现在有一件事需要我决定。→ §4 策展管线 + §4.6 四槽主界面。
2. **世界为角色命运服务，不是反过来。** 每个大系统问「会不会制造玩家与角色的关系变化」；世界事件必须落到个人命运。→ §4.1 翻译层。
3. **玩家要清楚自己的身份和权力边界。** → §3 **玩家=祖魂之声**（决策已定）。
4. **角色自治必须「意外但合理」，不是随机。** 每个意外要能追溯到人格/记忆/红线/关系/现实压力。→ §5 归因与门控（**代码强制**）。
5. **后果要分级，别一上来毁玩家心血。** 可恢复 / 高代价 / 不可逆三层；D0–D3 尽量只给前两层，不可逆需牵挂建立后开放。→ §4.3 后果分级闸。
6. **不要把编年史当主玩法。** 主界面=1 状态卡 + 1–3 高光卡 + 1 待决策 + 1 回响；长编年史只给深度玩家懒展开。→ §4.6 + §6.2 回响。
7. **世界要有密度，不是面积。** 优先「一个村子里 20 个关系复杂的人」，而非「2000 个名字」。→ §4.5 出生即 20 人关系网。
8. **付费不能改变命运胜负。** 可卖叙事密度/外观/槽位；不卖更快抢机会/更高成功率/反应速度。→ §7（结合已落地的 `engine/arbitration`）。
9. **必须设计「回来的一分钟」。** 60 秒回答 6 个问题。→ §6。
10. **世界会等待玩家，但不假装暂停。** 日常自治，命运收件箱——日常它自己活，真正伤筋动骨的命运节点系统推到玩家面前。→ §4.2 准入门槛。

## 2. 三个关键决策（本宪法新定的）

| # | 决策 | 一句话 |
|---|---|---|
| **D1 玩家身份** | **祖魂之声**（寄宿血脉的先祖意识） | 你是「她能听见、但不能强迫的一道声音」。抗命因此是**物理常态而非 bug**；直控=罕见的「灵魂共乘」。 |
| **D2 前台机制** | **单向策展管线**：Translator → FateScore → 三档路由 | 后台跑完整世界，前台只看「与她命运相关、且值得我决定」的东西。 |
| **D3 自治约束** | **代码强制的归因 + 门控意外** | 每个自治选择必带可解析的 cause；无因的戏剧性意外判 OOC 回退；突然恋爱/卖传家宝/叛变硬绑前因。 |

## 3. 玩家身份：祖魂之声（操作化原则 3）

**决策：在守护灵/命运顾问/附身者/祖先意识/脑中声音里，选「祖魂之声」**——以「祖先意识」为身份根、「守护灵之声」为日常交互态、「灵魂共乘」为直控特例。

**为什么**：五个候选里只有它同时满足三件事——① 身份本身就解释了「为什么我能干预却不能强迫」（祖魂给意志/记忆/情绪推力，落地要靠她那具有自己神经和恐惧的肉身，**所以抗命是物理常态**）；② 天然兼容血脉传承（祖魂随血脉延续，角色死亡接管后代不破设定）；③ 把直控降为「罕见、有代价的灵魂共乘」，正好做成原则 8 允许的养成奖励而非 P2W 捷径。（「附身者」被否决：暗示傀儡、抗命=故障会让玩家愤怒；「命运顾问」太外部解释不了直控；「脑中声音」太轻扛不起牵挂与传承。）

### 3.1 身份三句锚（写死进开场 10 秒第一屏）
- **我是什么**：「你是寄宿在【角色名】血脉里的一缕祖魂。她听得见你，但她有自己的命、自己的怕。」
- **我能做什么**：「你能给她方向，给她勇气，在她回望先祖时被她求问。」
- **我不能做什么**：「你不能替她迈步。她若不信、若太怕、若违了她的红线，她会自己选——那是她在长大。」

> 开场快照末尾强制弹一张**第一次抗命预埋卡**（她做了一件合理但你没料到的小事）+ 黄字「**这是她，不是你。**」——D0 第一屏就把「她会不听」设成已知规则，而非后期背叛。

### 3.2 全局 UI 语气与动词（求问/祈愿态，禁命令式）
内部机制（`doctrine`/`task`/`order` + `engine/decision` 路由）**全不动，只换皮换动词**：

| 内部 | 对外（祈愿层级） | 措辞 |
|---|---|---|
| doctrine | 家训 / 嘱托 | 「你愿她记住…」 |
| task | 指点 / 托梦 | 「你指点她去…」（非「命令她去」） |
| order | 叮嘱 / 疾呼 | 「你疾声叮嘱：别去！」 |

- 收件箱卡面主语是**她**：「阿采在桥头停下，回望先祖——她问你：『我该不该把刀交出去？』」选项前缀统一「你劝她… / 你叮嘱她… / 你由她…」。
- 回执**永不**出现「命令成功」，改「**她听进去了**」/「**她迟疑了，但还是…**」。
- GDD §5.1 共鸣度 → 玩家可见标签更名 **「回响 Echo」**（「她有多常听见你」）；底层=Loyalty/漂移三轴聚合视图不变。

### 3.3 抗命叙事化为「她的成长」
任何 `refused`/兜底（现有 `resolveDirectiveCompliance`/冻结清单路径）UI 层强制附一张**「她为什么没听你」溯源卡**，归因必须落到原则 4 的五来源之一并显式引用底层数据，末尾固定成长旁白「**她第一次没有照你说的做。她在变成她自己。**」，并写入角色编年史作为人物弧节点。

### 3.4 术语收敛（全仓库 UI 层）
废弃「化身 / 指挥官 / 主人 / 下达指令」；UI 一律「祖魂」（你）与「她/他」（角色）。`玩家魂` 保留为内部代号、UI 不可见。内部代码标识符（`doctrine/task/order/forcedOrder/activeImmediateOrderForUnit`）不改，仅约束呈现层。

## 4. 后台→前台策展管线（操作化原则 1/2/5/6/7/10）

**核心**：用一条**单向策展管线**解决「大世界 vs 开盒」的矛盾。后台无限 agent 照常跑，前台只看翻译机的输出。

```
后台完整世界(events流)
  → M1 Translator：世界事件 → 个人命运（相关性评分，≥0.35 才有资格）
  → M2 FateScore = 不可逆度 × 牵挂相关度 × 情绪强度
  → 三档路由：<0.25 自治不打扰 / 0.25–0.55 高光卡 / ≥0.55 待决策
  → M3 后果分级闸（牵挂等级 × 在世天数 硬锁不可逆）
  → M6 四槽主界面
```

### 4.1 M1 翻译层（纯代码，0 LLM）
对每个宏观事件 E 与玩家角色 P 计算
`Relevance(E,P) = 0.32·熟识 + 0.28·红线 + 0.18·目标 + 0.14·债仇爱 + 0.08·地理`（权重[待测试]）。
- **熟识**：E 涉及的人是否在 P 的关系表里，取关系强度归一。
- **红线**：是否触碰 P 离线宪章红线诱因。
- **目标**：是否改变 P 当前 L2 目标可达性（仇人出现/商路被断/盟友倒台）。
- **债仇爱**：是否落在 P 的债/仇/爱清单（§4.5 的 hooks）。
- **地理**：发生 region 与 P 当前 region 的图距衰减。

**阈值 0.35** 才进候选；低于=世界自己消化、玩家永不可见——把「无限事件」砍到「与她有关的几十件」。

**翻译模板**（data-driven `translation_templates`，把系统事实改写成对她的关系/压力变化）：
- 战争·征召：`target∈P.friends` → 「{friend} 被征召去 {front} 了」+ 置远 + 新增 worry 压力位 + `hook{separation}`。
- 贸易·欠债：`creditor∈P.acquaintance 且 P.debt>0` → 「{creditor} 放话三天不还就上门」+ 压力↑（债额大则升级命运节点）。
- 势力·投靠：P 因生存压力自评投靠了 `leader∈P 讨厌的人` → 「为了活命，她投到了 {你讨厌的人} 门下」（**必须可追溯到现实压力位，不可 AI 凭戏剧性触发**）。

### 4.2 M2 命运节点分（准入门槛，原则 10）
`FateScore = 不可逆度 × 牵挂相关度 × 情绪强度`，三档路由（与 `decision.Router` 的 Reflex/Deliberate 同构，新增 Curate 档）：

| FateScore | 路由 | 玩家体验 |
|---|---|---|
| < 0.25 | **自治不打扰** | 写 events，不进前台（仅懒展开编年史可见） |
| 0.25–0.55 | **高光卡** | 进主界面开盒槽，可一瞥 + 轻交互，不要求决策 |
| ≥ 0.55 | **待决策** | 升级成 `PENDING_DECISION`，带倒计时 + Copilot 2 选项 |

**硬规则**：触碰红线 / 施加不可逆后果的事件，无论分数一律强制进待决策。**每日预算**：进待决策的命运节点 ≤3/自然日（防轰炸），溢出降级高光卡。**过期兜底**：倒计时到按宪章兜底，但若结果与玩家倾向相反，事后必给一张「**你没回来，于是她自己选了——{结果}**」回响卡（**修正 §11.6「安全网杀死损失厌恶」**）。

### 4.3 M3 后果分级闸（原则 5）
`consequence_gate` 表，每层由牵挂等级 × 在世天数解锁：

| 层 | 内容 | 解锁条件 |
|---|---|---|
| **层1 可恢复**（D0 全开） | 受伤可治/欠债/错过机会/关系降温/短期分离/丢非 pinned 物 | 始终 |
| **层2 高代价** | 结仇/被迫离乡/背叛一个盟友/重伤致残/还不起的大债/暗恋移情 | 牵挂≥40 或 在世≥3 天（离乡/背叛盟友需牵挂≥50） |
| **层3 不可逆** | 死亡/生育/永久叛变/pinned 永久丢失/血脉断绝 | 牵挂≥70 **且** 在世≥7 天 **且** 已发生≥1 次层2（三条 AND） |

**D0–D3 把层3完全锁死**——物理上不会出现死亡/永久叛变。**可追溯断言**：任何层2/层3 后果落地前必须携带 `provenance` 链（指向人格轴/记忆/红线/关系/压力之一），否则降级层1或丢弃——「她叛变」必须能答出「因为 power 野心高 + 势力崩溃摘要进了她记忆 + 她欠你讨厌的人情」。

### 4.4 M4 牵挂等级 Attachment（可计算，复用 GDD 共鸣度）
`Attachment ∈ [0,100] = 100·sigmoid(0.35·共鸣度 + 0.25·在世天数项 + 0.20·主动回访项 + 0.20·共创项 − 偏置)`（系数[待测试]，新角色 D0≈10–15）。
- 主动回访只算「无推送主动打开」；共创项=亲手处理的命运节点数 + 直控高光次数（禀赋效应）。
- **Attachment 不可付费购买**（反 P2W）：付费买不到解锁不可逆后果的速度。

### 4.5 M5 出生即 20 人关系网（原则 7，密度而非面积）
出生时确定性生成「她的村子」（半径 1 region，20 人 NPC agent，全进后台自治），并给 P 预置关系网（FNV(worldID+seed) 派生，配额[待测试]）：

| 关系 | 数量 | 备注 |
|---|---|---|
| 家人 | 2–4 | 带「未说出口的期待」或「家庭债」 |
| 朋友 | 3–5 | ≥1 个**命运绑定密友**（战争/天灾优先落他身上） |
| 债主/被欠 | 1–2 | 出生即背债或被欠，立即制造压力位 |
| 暗恋/旧情 | 1 | 单向 affection 高、对方未知（秘密），潜在层2 移情 |
| 仇人/宿敌 | 1–2 | 带可追溯过往记忆种子 |
| 中性熟人 | 补满 20 | 构成关系网密度背景 |

强制写入：每段关系 ≥1 条 SeedMemory（解释「为什么」）+ ≥1 个 Secret（玩家知道、村里多数人不知道）+ 1 个绑定债/仇/爱的 LifeGoal（给原则 1「她想要什么」一个具体答案）。这 20 人不是名字列表：每人有 personality + 与 P 的关系 + 与其他村民 ≥1 条关系。

### 4.6 M6 四槽主界面（原则 6）
固定四槽，长编年史懒展开、不当主玩法：

| 槽 | 内容 | 来源/排序 |
|---|---|---|
| **A 状态卡**（顶部一行） | 离线时长/生死/HP/地名/待决策数/紧急数 | 永远第一屏、零交互可读完（回答原则 9） |
| **B 高光卡 1–3 张**（开盒区，core toy） | 每张 ≤30 字旁白 + 情绪色 + 轻交互三选 | FateScore 0.25–0.55 的事件，排序 0.6·FateScore+0.4·新鲜 |
| **C 待决策 1 张**（命运节点） | 情境 + 倒计时 + Copilot 2 选项（标后果分级）+ 不处理会怎样 | FateScore≥0.55 队首 |
| **D 回响 1 条**（原则 6 点名） | 「因为你让她{X}，这次她{Y}」+ 关联前次决策 ID | 上次玩家决策的下游后果（§6.2 Echo） |

排序总原则：紧急>情绪>新鲜；任何时刻屏幕 5 秒内可读完。

## 5. 意外但合理：归因与门控（操作化原则 4，**代码强制**）

**每个自治选择必须随意图同时产出一个结构化「归因」**，否则判 OOC 回退——「意外」由数据决定，而非 LLM 觉得有戏剧性。

### 5.1 归因结构（扩展 `unitDecisionPayload`）
```
Attribution{
  Primary    CauseRef          // 必填
  Supporting []CauseRef        // 0–2 条
  SurpriseLevel int            // 0–3
  NarrativeZH string           // ≤40字给玩家看的因果句
}
CauseRef{ Kind, RefID, Weight(0–1), SnippetZH }
```
`Kind` 严格 6 类，每类 `RefID` 必须能在数据层解析到真实记录：
- **persona_trait** → 8 维 personality | 6 维 Ambition，且 `|value−0.5|≥0.25` 才可作 primary；
- **memory** → memoryRow.ID，且 `Importance≥6 或 |EmotionWeight|≥0.5` 且 salience>0.15；
- **redline** → 离线宪章红线条目 ID；
- **relation** → 目标 unitID，且某关系轴 `|value|≥0.3`；
- **pressure** → {hunger,threat,debt,injury,fatigue} 且越过 L1 护栏阈值；
- **order_echo** → 上一条玩家 directive/接管的 event_id（回响用）。

### 5.2 归因校验器（纯代码，已落地，见 §9）
`ValidateAttribution(attr, snapshot)` 硬判定（任一不过→OOC）：① primary 解析成功；② 越过显著阈值；③ `SurpriseLevel≥2` 时 primary/supporting 必须有一条 `kind∈{memory,relation,redline} 且 weight≥0.5`（纯 persona 不足以撑「明显意外」，否则一切都能用「她性格如此」搪塞）；④ 门控动作走更严的 GateCheck。**回退**：写 `OOC_REJECTED` 审计事件、改走 `safeFallback`。目标 OOC 回退率 <8%（>15% 说明前因没喂够或阈值过严）。

### 5.3 门控意外表（突然恋爱/卖传家宝/叛变 硬绑前因）
某些 action 即使归因「看起来合理」也必须命中专属前因，否则**连选项都不生成**：
- **romance**（突然恋爱）→ 对目标 ≥2 条 relation/event 记忆 且 affection≥0.4 且跨 ≥3 个自治窗口累积。缺失则候选剔除。
- **sell_pinned**（卖传家宝）→ 必须有 `pressure=debt/threat` 作 primary，且非 Permanent 锚定物；否则进冻结清单交玩家，绝不自治执行。
- **defect**（突然叛变）→ 复用 `resolveDirectiveCompliance` 的 refused 出口，额外要求 `Loyalty<阈` 且（负面 memory 或 relation 恶化 或 势力衰退压力）≥1 条；纯 `Ambition.freedom` 高不足以单独触发。

> **关键**：门控只在前因未建立时「禁止」；一旦玩家自己用接管/直控制造了前因（如强迫她杀人→写下负面 memory），门控自动解锁——与原则 5「不可逆后果牵挂建立后才开放」天然咬合（不可逆门控动作还要叠加牵挂解锁线）。

### 5.4 把「因」暴露给玩家
高光卡/待决策统一渲染 `NarrativeZH` 因果句：默认只展示 primary 一条（「{角色}{动作}，因为{primary.snippet}」≤40 字，情绪色由记忆 EmotionWeight 符号决定）；supporting 折叠进「为什么？」；`kind=memory` 时给「回到那一刻」链接跳原始 event 链（复用 `chronicle_entries.source_event_ids` 懒渲染）。轻交互〔意料之中/有点意外但合理/太离谱〕回写 surprise 校准 = GDD §8 惊喜命中率采集口。

## 6. 回来的一分钟（操作化原则 9）

固定一块上线首屏，6 问各绑一个槽位 + 文案模板 + 数据源，60 秒读完：

| 问 | 槽位 | 数据源 |
|---|---|---|
| Q1 她还活着吗 | 状态卡·生命徽章 | `LifeState/HP/LivesRemaining`（已死→死亡传记卡+血脉接管入口） |
| Q2 她在哪 | 状态卡·地点行 | `RegionID`→region 名 |
| Q3 发生了什么大事 | 1–3 高光卡 | 离线期 importance≥7 events，按 importance×emotion 取 top1–3 |
| Q4 跟我上次选择什么关系 | **回响带**（贯穿条） | Echo 机制（§6.2） |
| Q5 我必须处理什么 | 待决策首条 | 未 resolved 的 PENDING_DECISION，按紧急×不可逆×情绪排序 |
| Q6 不处理会怎样 | 首条决策倒计时 + 兜底预告 | 兜底动作 + 后果分级标签 |

**硬规则**：Q5 首条决策永远 ∈ {可恢复, 高代价} 层；**不可逆决策绝不进倒计时兜底**，必须显式停在玩家手里。

### 6.2 回响 Echo（「因为你上次选了 A，所以这次变成 B」）
`EchoCard{ prior_choice_zh, prior_event_id, consequence_zh, consequence_event_id, valence }`。触发：本次离线期存在 attribution 链中 `kind=order_echo` 指向上次玩家 directive/接管 event 的事件，或上次 `player_intervention` 记忆成为本次自治选择的 primary memory 因。
- 正向：「上次你让她**放过了那个逃兵**，这次他在你们被围时**回来替她挡了一刀**。」（共鸣度 +，提示「你越来越懂她了」）
- 负向：「上次你强迫她**杀了求饶的俘虏**，这次面对你的命令，她第一次**犹豫了**。」
- **约束**：`order_echo.ref_id` 必须真指向存在的玩家动作 event，无则不生成（禁止编造「因为你上次」——这也是一种 OOC）。

## 7. 付费不改命运胜负（操作化原则 8，结合已落地引擎）

三道闸把「成功率/反应速度」挡在付费之外：
1. **关键节点成功率** → 由 `engine/arbitration.Resolve` 确定性结算：胜率=score_i/Σscore，`dedupMaxScore` 保证高频只算一次、对入队顺序不变。**付费档不进入 score**（score 只由角色投入/属性/距离构成）。
2. **反应速度** → 仲裁 tick 统一节奏，付费高频只让意图「更早排队」不决定谁赢；应激插队**账号级**封顶。
3. **归因校验与门控对所有付费档一视同仁**——阈值不因付费放宽；**免费角色关键节点有「决策质量地板」**（临时升档同级模型，成本由「<2% 上 LLM」吸收）。

| ✅ 可卖（叙事密度/外观/便利） | 🚫 不卖（命运胜负） |
|---|---|
| 更精美的高光卡/死亡传记卡、更漂亮的高光演出与 Echo 动画 | 关键节点成功率 |
| 更多角色槽 | 反应/插队速度 |
| 更深的回忆整理（记忆容量×2/反思更深） | 归因校验宽松度 |
| 多视角历史解读、归因「为什么？」更长追溯 | 门控/不可逆后果解锁速度 |

**红线判据（可证伪）**：任一付费点若能在两角色争夺同一排他结果时改变 `arbitration` 胜负 → 判脏下线。蒙特卡洛：付费方设 5×频率+插队+强模型跑 1e4 次，关键节点胜率须 ≈score 比例（±2%），>60% 即判 P2W。**新增商业红线：不卖牵挂、不卖后果解锁速度。**

## 9. 已落地：归因校验引擎（让「意外但合理」是代码、不是纸面）

为把原则 4 从软约束变成**硬执行**，已在 `backend/internal/engine/decision/attribution.go` 实现纯逻辑、可测试的归因校验，并补了测试（`go test ./internal/engine/...` 全绿）：

| 能力 | 函数 | 作用 |
|---|---|---|
| 归因结构 | `Attribution` / `CauseRef` / `CauseKind` | 6 类 cause 的结构化表达 |
| 归因校验 | `ValidateAttribution(attr, snap) Verdict` | 解析+显著阈值+「明显意外需强前因」+ snippet 存在性，任一不过判 OOC，给确定原因码 |
| 门控意外 | `GateSurprise(action, snap) GateResult` | 突然恋爱/卖传家宝/叛变 的硬前因谓词，缺前因→剔除候选/转冻结 |

它直接扩展了上一轮的 `engine/decision` 包，与 `engine/arbitration`（反 P2W）一起，构成「**决策用 LLM、结算用代码**」中**结算层确定性**的两块基石。

**已接入 `session.generateUnitDecision` 并默认开启强制**：决策载荷含可选 `attribution` 字段；决策前 `prepareAttribution` 构造完整快照——人格维(persona_trait) + 现实压力位(pressure) + **实时记忆(memory，并把可引用的记忆 ID 暴露进 prompt)** + **对外关系四轴(relation，归一到 [-1,1])**，使这四类前因都能被解析；prompt 明确要求 LLM 只用这四类、`ref_id` 取自列表、宁可省略也不编造。每次带归因的决策经 `ValidateAttribution` 校验并计入 OOC 遥测（`Service.AttributionStats()`）；线上经 `httpapi/router.go` 的 `SetAttributionEnforcement(true)` 开启强制——**无源戏剧性归因(SNIPPET_FABRICATED / SURPRISE_UNGROUNDED / CAUSE_* 等)优雅回退到安全决策**（继续按兵不动，非乱码困惑态）。`redline/order_echo` 两类 ref 仍待离线宪章/回响功能落地后接入；若线上 OOC 率过高可一键改回影子模式（`SetAttributionEnforcement(false)`）。
