# 共享世界改造方案（方向 B）— 2026-06-10

> 把游戏从「每玩家独立 session 世界副本」改为**真·共享世界**（所有玩家在同一世界实例、能相遇、共享 boss/NPC/编年史，取消"主角"绝对概念）。
> 本文由 6-agent 深度研究（5 子系统并行 + 综合）产出，**全部结论逐点核实代码证据**。是后续分阶段实施的规格。

---

## 0. 核心结论：这是「改造接线」，不是「重写」

**最反直觉、也最关键的发现**：项目早就为共享世界铺了一整套骨架，而且**测试齐全、单条链路（世界 boss）端到端真跑通**。真共享世界缺的不是表、不是端点，**只缺三件硬缺口**：

| 缺口 | 现状证据 | 后果 |
|---|---|---|
| **A. 几何私有** | 每玩家降生走 `GenerateWorld(seed=now.UnixNano())`（mainworld.go:209）各自生成 7 区地图存进 `state.Zones` blob；`world_maps/world_tiles` 表存在但降生不用 | 玩家挂同一 `world_default` 标签，但物理上在各自独立地图里 |
| **B. region_id == sessionID** | `ambient_scheduling.go:79` 默认 `regionID := sessionID`，仅分片 flag 开才升级 | 两玩家即使同 world，永不同 region、永不相遇 |
| **C. 单位严格 session 私有** | `loadSession→units.ListBySession`，`buildSnapshot` 永远看不到别玩家角色；缺 `idx_units_world_region` 索引 | 同 world 角色互相看不见、不在同一快照 |

**工作量**：MVP（共享几何只读 + 玩家相遇 + 事件层零改）约 **2-3 周**；完整方向 B（统一世界推进 + per-tile 仲裁 + 私有档迁移 + 主角解耦）约 **1.5-2.5 个月**。

**警示**：别被"有代码"误导成"在跑"。线上默认只有 `WORLD_BINDING=shared` 一层真通电，`REGION_SHARDING/RUNNER_APPLY/AUTO_MATCH/FATE_AUTOTICK` 全默认关——大部分跨玩家能力从未被真实玩家触发过。

## 1. 已建成可直接复用（地基大头，接上即用）

全部有代码 + 多数有测试：

- **`cross_events` append-only 总线 + `worldbus`**：权威序 `(world_tick,occurred_at,id)`、无外键跨分片、只追加可复算——共享编年史/「谁先动手」仲裁的现成事实源。
- **`world.AdvanceTick` + `worlds` 表**：共享世界权威推进节拍，双驱动原子发号。
- **`world_members` + `world.Join`（幂等）**：现成「谁在这世界」花名册。
- **`world_chronicle` + `/api/worlds/:worldId/chronicle`**：群像史已按 `world_id` 共享、**本就非主角口吻**（直接解决你上一个问题）。
- **`world_bosses` strikeTx**（原子扣血+账本+闩锁结算）：一切共享并发写同一对象的模板，唯一端到端跑通的跨玩家完整链。
- **`social_objects` + `MatchScore` 四因子 + `consent_gate` 三档异步同意 + `seven_interactions` + `arbitration.Resolve`**：跨玩家社交协议逻辑完整确定性，**只缺候选池跨 session**。
- **`region/registry`（HOT/WARM/COLD 活跃度 + 威胁等级 + per-region 时钟）+ `agentqueue`（持久唤醒队列）+ `region_leases`（单写者租约）+ `regionrunner` 引擎**：**最值钱资产**——现成的「不绑 session 的世界级跨玩家执行器」，已按 region 而非 session 调度。
- **`units` 三列（world_id/region_id/last_active_tick）+ version 乐观并发列 + `SetUnitScope`/`ListActiveByRegion` + `Mutator.ApplyOptimistic`**：共享世界角色落库与防覆盖底座。
- **切片器 `currentZone`/`setCurrentZone`/`zoneVisibleUnits`/`reprojectCurrentZone`**（阶段1 已做）：「世界→玩家视角」现成抽象——**迁移成本集中在数据来源，不在切片逻辑**。
- **`world_maps`/`world_tiles` + `SaveMap`/`LoadLatestMapSnapshot`**：共享几何现成持久化，只是降生没用它。
- **`LifecycleClass` 三分类 + aging 永生双保险**（阶段4）：支撑「取消主角绝对概念但保留你操作的那个」。

## 2. 目标架构

**世界态权威外移到 `world_*` 表 + 玩家角色按 `world_id/region_id` 落库 + 视角快照按地理区切片 + 中心化 world-runner 统一节拍 + 共享事件总线群像化。**

- **世界状态权威**：`EnsureDefaultWorld` 时一次性 `GenerateWorld(固定 RegionSeed)` + `SaveMap` 到 `world_maps/world_tiles`；`state.Map/Zones` 降为「当前区投影缓存」（`reprojectCurrentZone` 机制已支持从权威源重投影——**最值钱的复用**）。`region_id` 从「调度分片桶」改为「真·地理子区（=ZoneID）」。
- **玩家角色挂载**：降生后对 hero 显式 `SetUnitScope(world_default_shared, 真实ZoneID)`（绕 flag gate）。账号→world 角色幂等模型（`account_id,world_id` 唯一索引）对方向 B 完全适用。
- **视角快照**：`buildSnapshot` 单位来源从 `ListBySession` 改为 `ListActiveByRegion(当前ZoneID)`——拉同区所有玩家+NPC。`zoneVisibleUnits` 仍按 CurrentZoneID 过滤。
- **世界推进**：中心化 world-runner（复用 `regionrunner` 整套引擎），改三点：region_id→空间锚、Apply 从 ambient 四动作升级为命运决策链、默认开 flag。世界时钟统一用 `world.AdvanceTick`。
- **并发裁决**：玩家直驱从「直改」改为「争夺入队、仲裁 tick 裁决」（`arbitration.Resolve` 胜率∝Score、付费不进、频率无关）；per-zone 用 `region_leases` 单写者串行；共享对象并发写复用 world_boss strikeTx 闩锁；单 unit 多写者用 `Mutator.ApplyOptimistic`（version 列）。

## 3. "主角"概念怎么重定义

**从「session 唯一绝对主角」→「世界众多角色之一，但仍是你账号操作的那个」。** 务实保留核心不变量：

- **不推翻** `LifecycleClass` 三分类与 aging 永生双保险（`isProtagonistUnit` 已是 unit 级判定、跨 session 正确）——玩家角色仍永生、不衰老死亡。
- **只改归属语义**：① `guardPlayerAction` 的「归属本玩家」门从「PlayerUnitIDs 含此 unit」改为「查 world_members/账号归属」（你只能操作自己降生的角色，但同一 world 可承载多个玩家角色）；② `mainWorldCharacterView` 从「取 PlayerUnitIDs[0]」改为「按账号取其归属的 protagonist」。
- **关键变化**：**别玩家的 protagonist 在我的快照里就是普通在场角色**（上图、可遇见、可交互），不是不可见的 NPC，也不享有「仅对我特殊」的待遇。"主角"成了相对每个玩家视角的「我的角色」。
- **命运开盒**：命运 beat 从「仅主角」守门改为「所有 active 的 protagonist」；修 cross-session 投递传 owner 自己 sessionID 的串味 seam（避免用 A 的红线锚判 B 的事件致 OOC 误判）。
- **编年史**：**无需改**——`world_chronicle` 本就是群像史，前端命运客户端的「世界编年史」呈现 N 个玩家角色交织的史诗。

## 4. 分阶段迁移路线

原则：**先共享事件层（已通零改）→ 共享几何只读 → 玩家相遇 → 共享进度 → 统一推进 → 私有档迁移**。每阶段独立可验收，新老用 `world_id` 区分世代。**最先别碰**：把多玩家塞进同一 session `state_json`（巨型行死穴）。

| 阶段 | 内容 | 验收 |
|---|---|---|
| **Phase 0 · 事件层**（已通，补遗）| 前端接 `/api/worlds/:worldId/chronicle` 露群像史；补 cross-session 投递 sessionID seam | 两真人玩家的角色之死/世界 boss 击败出现在同一编年史；A 死讯扇出到 B 的关心者收件箱 |
| **Phase 1 · 共享几何（只读）·MVP** | `EnsureDefaultWorld` 填固定 RegionSeed + GenerateWorld+SaveMap；降生改 `LoadLatestMapSnapshot`；新建 `world_default_shared` 世代（旧私有档保留可玩，**不强制迁移**）| 两个新降生玩家拿到逐格相同的 7 区地图/POI/boss 刷新点 |
| **Phase 2 · 玩家相遇·MVP** | 降生后 `SetUnitScope(真实ZoneID)`；建 `idx_units_world_region`；`buildSnapshot` 改 `ListActiveByRegion`；同格冲突跨玩家算 | 玩家 A 在 zone_neutral_start 看见同区玩家 B 角色、能发起七交互 |
| **Phase 3 · 跨玩家撮合/交互** | `buildMatchCandidates` 输入改跨 session；seven_interactions 从 ops 手动改玩法内自动 | 真人 A、B 进同一撮合候选池，走零和裁决 |
| **Phase 4 · 共享进度** | zone boss/任务/解锁从 per-session 私有改 world 级共享（某玩家打掉 boss 全区生效），仲裁 tick 串行结算 | A 击败 zone boss 后 B 看到已击败、共享战利品按贡献分配 |
| **Phase 5 · 统一世界推进** | 中心化 world-runner 接管全体玩家角色，**停掉** `RunFateAutoTickLoop` 对共享 session 的推进（二选一防双推进者）；Apply 升级命运决策链 | 所有玩家角色在同一世界 tick 节拍自治、离线也活 |
| **Phase 6 · 私有档迁移**（可选/谨慎）| 旧私有副本坐标重映射+去重——**建议不强制**，旧档作"单机怀旧档"保留 | — |

**先决工程债（Phase 1 前或并行）**：① 切 **MySQL**（SQLite `SetMaxOpenConns(1)` + 巨型 state_json 单行是死穴，多玩家挤进 world_default 前必须切）；② 所有玩家直驱写路径切 `Mutator.ApplyOptimistic`（乐观并发，否则多玩家同操丢更新）；③ 统一 `worlds.tick` 与 region-runner 墙钟时钟源。

## 5. 头号风险

1. **误判性质**（头号）：把"缺骨架"误判为需重写——骨架已建成+测试齐全，重写会丢大量已绿测试与硬不变量（Mutator/确定性/反 P2W 仲裁）。坚持「改造接线」。
2. **region_id 语义二义性**（隐形地雷）：大量代码按 region_id 写得"貌似 region-scoped 实则 session-scoped"，贸然改真地理子区会行为突变（ambient 跨玩家混跑/lease 争抢/空 RegionID 单位永久饿死）。规避：逐一审计"region_id 空→回退 sessionID"兼容分支；新世代隔离；Phase 2 先只读上图。
3. **巨型 state_json 死穴**：多玩家塞一个 session state_json 会变 GB 级热点行。规避：共享几何必须外移到 `world_maps/units` 表，绝不走 session blob；Phase 1 前切 MySQL。
4. **并发丢更新**：units 全量 status_json Save 无单位级锁；`guardPlayerAction` 是 per-session 锁，两玩家动同一 tile 各持各自锁互不阻塞。规避：写路径切 `ApplyOptimistic`；争夺收敛到仲裁 tick；per-zone 用 `region_leases`。
5. **双推进者冲突 + 45min 异步长事务**：world-runner 与 `RunFateAutoTickLoop` 并存会双推同一单位；一区长执行期间该区玩家全 409 冻结。规避：Phase 5 二选一；执行粒度从「整 session 一轮」降到「单决策点/单 region 一拍」。
6. **跨玩家硬红线被破坏**：「B 永远只能写一条 cross_event，改不了 A 的 units/relations」。**Phase 3 状态（2026-06-10 评审修复后）**：① **关系写红线已真实武装**——consent accept 改为只写**接受方自己 owner 一侧**出边（`source=target=接受方 → target=actor=发起方`），写前经 `assertOwnSideRelationWrite` 钉死 `source==接受方`（方向写反即拒 + 落 `CROSS_WRITE_DENIED` 审计），消除了原先「接受方 session 直写发起方 outgoing relations」的张力；② **units 整记录写断言**（`SaveOwnedBy`/`assertOwnSideUnitWrite`/`saveOwnSideUnit`）是**为 Phase 4「共享 NPC / 整记录写」预置的纵深护栏**，Phase 3 当前**确无** `units.Save(peer)` 调用点（跨玩家路径只写本侧 relations + append cross_event，是设计纪律的结果，非缺口）。规避（已落地）：关系写经 own-side 断言；units 写预置断言 + 测试守「跨玩家路径不裸 `units.Save(peer)`」。
7. **主角概念耦合点多**（`PlayerUnitIDs` 14 处 load-bearing）：改"session 唯一主角"牵一发动全身。规避：不推翻三分类/永生双保险，只改归属语义，全量梳理 14 处。
8. **world_boss 已知残留在并发放大下变大**：strike 端点无鉴权/幂等（贡献伪造/刷）、结算未与闩锁同事务。规避：并发放大前补 strike 鉴权/幂等 + 结算事务化。

### 5.1 Phase 3 部署/运维须知（consent 链路）

- **离线方 consent 不会永挂**：层2/3 pending 的「超时失效 + 层3 给发起方投回响卡」由**独立后台 sweep**（`RunConsentExpirySweepLoop`，**恒开、无 flag 门控**，默认 10min 一拍，`QUNXIANG_CONSENT_SWEEP_SECONDS` 可调）驱动，**不依赖** `QUNXIANG_FATE_AUTOTICK`、**不依赖**离线方是否上线。该 sweep 只做全局安全操作（置 `expired` + 写发起方自己收件箱），绝不写任何他人 units/relations、不跑 LLM。
- **层2「据归因自治接受」仍由 target 自己 session 边界驱动**（`settleConsentsAtBoundary`，作用域=target 所属 session）：它会写**接受方本侧出边**，必须由本侧驱动以守跨玩家硬不变量。若希望真离线方也能「据归因自治接受」（而非仅超时失效），生产可同开 `QUNXIANG_FATE_AUTOTICK`（后台 ticker 周期推进活跃主世界 session 含离线方的边界）；不开时离线方层2 pending 走 sweep 的 72h 超时失效兜底（不应用关系效果），不会半瘫。
- **玩家自处理 consent 是玩家权限**：玩家档路由 `GET /api/fate/consent/pending/:unitId`、`POST /api/fate/consent/:reqId/resolve`（按账号→角色归属鉴权，强制 Bearer）供前端「来意」面板用；配 `QUNXIANG_OPS_TOKEN` 后玩家仍可处理自己角色名下的 pending。旧 `/api/consent/*`（ops-gated）保留作运营巡检/兜底。

## 6. 建议起点

**强烈建议从 MVP（Phase 0+1+2）开始**——这是"玩家能在同一张图上相遇 + 共享世界几何 + 群像编年史"的最小闭环，约 2-3 周，且：
- 用 `world_default_shared` 新世代隔离，**旧私有档零影响**（继续单机玩）。
- 不碰最危险的并发写裁决（Phase 2 先只读上图：能看见、能社交，但进度仍私有）。
- 验证"真共享世界"体验后，再决定是否推进 Phase 3-6（共享进度/统一推进）这个 1.5-2.5 月的深水区。

**但 MVP 有一个硬前提**：Phase 1 把玩家挤进 world_default 前**必须先切 MySQL**（SQLite 单连接 + 巨型 state_json 撑不住多玩家共享）。这是运维/部署决策，需要你确认数据库方向。
