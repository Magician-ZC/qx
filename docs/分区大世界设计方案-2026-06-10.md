# 一念 · 分区大世界设计方案（2026-06-10）

> 把游戏从「单图单会话」升级为**魔兽式分区大世界**的总设计。
> 用户愿景（原话归纳）：一个世界由多片区域地图组成，每个区域有独立的 boss/副本/随机 boss/任务；
> 玩家通过任务去往不同区域；怪物有等级；玩家角色与固定功能性 NPC 不衰老死亡、其余 NPC 会衰老死亡；
> 整个游戏世界要有编年史。
>
> 本文是**规格说明**——后续分阶段开发以此为准。设计原则承袭 CLAUDE.md：
> 决策用 LLM、结算用代码；受保护字段经 `status.Mutator`；确定性随机（FNV）；新能力 flag 渐进；中文注释。

## 0. 一句话蓝图

```
世界 World
 ├── 区域 Zone A  ·  自由阵营主城区   一张地图 + 主城 + Lv1-10 怪 + 区域boss + 副本 + 任务线
 ├── 区域 Zone B  ·  秩序阵营区       一张地图 + 中城/村庄 + Lv10-20 怪 + 世界boss + 任务线

导航：世界地图(看全部区域,点击前往) + 区域地图(当前区域全屏hex) + 小地图(她周边)

```

## 1. 世界与区域的数据结构（核心改造）

### 1.1 落地方案：session 扩展为「世界含多区域」

**现状**：一个玩家 = 一个 session = 一张 `state.Map`（24×16）。所有逻辑（渲染/移动/采集/POI/威胁）直接读 `state.Map`。

**改造**：把 session 的"单张地图"升级为"世界含多个区域"。角色状态（背包/装备/属性/关系/记忆）天然连续——**不跨 session 迁移角色**，只在同一 session 内切换"当前区域"。

```go
// state.go（扩展）
type State struct {
    // ... 既有字段 ...
    Zones          []Zone   `json:"zones,omitempty"`           // 世界的全部区域（空=旧单图存档，向后兼容）
    // Map 字段保留：作为「当前区域地图」的兼容投影（旧代码读 state.Map 仍工作——指向 current zone 的 map）
}

type Zone struct {
    ID           string             `json:"id"`              // 区域 id（zone_freedom_capital 等）
    Name         string             `json:"name"`            // 「晨曦平原」「铁律城郊」…
    FactionID    string             `json:"faction_id"`      // 阵营归属（freedom/order/chaos/neutral）
    Kind         ZoneKind           `json:"kind"`            // capital(主城区)/town(中城)/wild(野外)/starter(新手)
    LevelMin     int                `json:"level_min"`       // 区域等级带下限（怪物等级范围）
    LevelMax     int                `json:"level_max"`       // 上限
    Map          world.MapSnapshot  `json:"map"`             // 该区域一张 hex 地图（复用现有 generator）
    Portals      []ZonePortal       `json:"portals"`         // 通往其他区域的传送门/边界口
    Settlements  []string           `json:"settlements"`     // 本区城镇坐标（"q,r"）
    BossThreatID string             `json:"boss_threat_id"`  // 区域 boss（复用 threat）
    DungeonIDs   []string           `json:"dungeon_ids"`     // 本区副本入口
}

type ZonePortal struct {
    AtCoord    string `json:"at_coord"`     // 本区出口坐标 "q,r"（通常在城镇/边界）
    ToZoneID   string `json:"to_zone_id"`   // 目标区域
    ToCoord    string `json:"to_coord"`     // 落点坐标
    Kind       string `json:"kind"`         // portal(传送门,需解锁) / border(边界,走到即过)
    UnlockTip  string `json:"unlock_tip"`   // 未解锁时的中文提示
}
```

**角色当前区域**：
```go
// unit.Status 或 Profile（扩展）
CurrentZoneID string `json:"current_zone_id,omitempty"` // 她此刻在哪个区域（空=主区/兼容旧档）
```

### 1.2 兼容与迁移
- `Zones` 为空的旧存档：视为"单区世界"，`state.Map` 即唯一区域，所有现有逻辑零改动继续工作。
- 新建会话：世界生成器产出 `Zones[]`，`state.Map` 投影为出生区地图，角色 `CurrentZoneID` = 出生区。
- 读"当前区域地图"统一走 helper `currentZoneMap(state)`（旧代码 `state.Map` 由该投影维护，渐进替换调用点）。

### 1.3 为什么不做"真无限"
真无限地图要重写世界存储（整块快照→chunk流式）、确定性、寻路、渲染，以周计且打破现有确定性/存档/战棋复用，收益对"命运沙盘"存疑。**分区世界用有限的多张图达到"跑很久到不了边、内容分区丰富"的魔兽式体验**，是务实且能落地的路径。

## 2. 分区世界生成

新文件 `internal/world/worldgen.go`（在现有 `generator.go` 单图生成之上的世界编排层）：

- `GenerateWorld(seed, factionLayout) []Zone`：确定性生成 N 个区域。
  - 默认布局：1 个**中立新手区**（出生）+ 三阵营各 1 个**主城区** + 若干**野外/中城区**（共 6-8 区，可配）。
  - 每区调 `world.GenerateMap` 生成地形图（复用现有），尺寸可按区域类型差异化（主城区大、野外区中等）。
  - 区域等级带递进：新手区 Lv1-5 → 阵营外围 Lv5-15 → 阵营核心/野外 Lv15-30。
  - 区域间用 `ZonePortal` 连成图（新手区 → 三阵营外围 → 各自核心；阵营核心之间为"边境/战场"区）。
- 阵营城镇体系（取代现有"城镇是无归属地形"）：
  - 每个**主城区**放 1 座**主城**（`TerrainCity`，阵营核心，传送枢纽）。
  - 每个阵营区放若干**中城/村庄**（`TerrainVillage`），绑定阵营归属。
  - 中立区放中立集市/新手村。
  - 城镇落 `Zone.Settlements` + 阵营标记，供传送门、功能性 NPC、任务发布锚定。

## 3. 怪物等级系统（`unit.Level` 已存在，扩展用法）

- **怪物等级**：野外怪/区域 boss 的 `unit.Level` 由所在**区域等级带**确定性派生（`FNV(zoneID, coord) → [LevelMin, LevelMax]`）。
- **等级影响**（结算用代码）：
  - 战力缩放：`scaledElite`（threat.go 已有）按等级放大攻防/HP。
  - 战利品/经验：高等级怪掉落更好（`engine/encounter.AllocateLoot` 已按贡献分赃，叠加等级系数）。
  - 玩家角色也有等级（`unit.Level`），击败怪物得经验升级（新增轻量经验/升级规则，纯代码结算经 Mutator 留痕——经验非受保护字段，升级带来的属性提升走派生属性重算）。
- **等级护栏**：低级角色进高级区域，怪物显著强（region 等级带 > 角色等级时遭遇威胁提示"此地凶险"），引导玩家按等级带推进——魔兽式分级体验。

## 4. 区域内容（复用现有 PvE 引擎）

| 内容 | 复用 | 接法 |
|---|---|---|
| 区域 boss | `threat.go` `TriggerEliteEncounter`/`ResolveEliteEncounter` | 每区 `BossThreatID`，在区域 boss 坐标触发 |
| 随机世界 boss | `threat.go` `TriggerFieldBoss` + `threat_refresh.go` 刷新 | 区域内确定性周期刷新（flag `QUNXIANG_WORLD_BOSS_AUTO`），多人协作消耗战 |
| 副本 | `dungeon.go` `RunDungeonForSession` | 区域城镇/秘境放副本入口，前端"进入副本"接已有端点 |
| 野外 PvE 威胁 | `engine/encounter` + 地块 POI 埋伏（已接） | 区域等级带决定威胁强度 |
| 战利品/惩罚 | `engine/encounter` `AllocateLoot`/`DegradePenalty` | 已通，按等级叠系数 |

**前端缺口**：这些后端能力多数 flag 关 + 没接命运客户端入口。分区世界落地时，区域地图上的 boss/副本/世界boss 标记 → 点击"挑战/进入"接既有端点（这也补上了之前 A 类未做透的"命运客户端 PvE 入口"）。

## 5. 任务系统（全新）

魔兽核心：去不同区域做任务。新文件 `internal/session/quest.go` + `quest_catalog.go`。

### 5.1 数据结构
```go
type Quest struct {
    ID          string       `json:"id"`
    Title       string       `json:"title"`        // 「肃清晨曦平原的野狼」
    GiverUnitID string       `json:"giver_unit_id"`// 发布任务的功能性 NPC
    ZoneID      string       `json:"zone_id"`      // 任务所在/指向区域
    Type        QuestType    `json:"type"`         // slay(讨伐)/collect(收集)/escort(护送)/explore(探索)/story(剧情)
    Objectives  []Objective  `json:"objectives"`   // 目标(杀N只X / 收集N个Y / 到达Z / 击败boss)
    Rewards     QuestReward  `json:"rewards"`      // 金币/物品/经验/声望/解锁传送
    Prereq      []string     `json:"prereq"`       // 前置任务(任务链)
    State       QuestState   `json:"state"`        // available/active/completed/turned_in
}

type Objective struct {
    Kind     string `json:"kind"`      // slay_kind / collect_item / reach_coord / defeat_boss
    Target   string `json:"target"`    // 怪类型 / 物品id / "zoneId:q,r" / threatId
    Required int    `json:"required"`
    Current  int    `json:"current"`
}

type QuestReward struct {
    Wallet      int      `json:"wallet"`
    ItemGrants  []string `json:"item_grants"`
    Exp         int      `json:"exp"`
    Reputation  int      `json:"reputation"`   // 阵营声望
    UnlockZone  string   `json:"unlock_zone"`  // 完成解锁某区域传送
}
```

### 5.2 玩家角色状态（扩展）
```go
ActiveQuests    []Quest  // 进行中（随 state JSON 持久化于角色或 session）
CompletedQuests []string // 已完成任务 id（任务链前置判定）
```

### 5.3 流程
1. 城镇功能性 NPC 发布任务（任务目录 `quest_catalog.go` 确定性投放，按区域/等级）。
2. 玩家接取（端点 `POST /api/sessions/:id/units/:unitId/quests/:questId/accept`）。
3. 目标进度由**世界事件驱动结算**（代码）：击杀怪/采集物/到达区域时，匹配 active quest 的 objective 递增（复用既有结算回调，不引入轮询）。
4. 完成 → 回城镇交付（`turn-in` 端点）→ 发奖（钱包/物品经 Mutator 与背包路径，经验升级，声望，解锁传送）。
5. **与命运层结合**：active quest 作为"她的目标"喂进自治决策上下文（`directive_context.go`）——离线时她会朝任务目标自治行动，上线你可直接推进。任务完成/失败入命运 feed 与世界编年史。

### 5.4 任务类型落地
- **讨伐 slay**：杀 N 只某等级怪 / 击败区域 boss（接 threat）。
- **收集 collect**：采集 N 个材料（接地块 gather）/ 交易获得。
- **护送 escort**：护送功能性 NPC 到目标区域（移动 + 沿途遭遇）。
- **探索 explore**：到达某区域/地标（接区域切换 + 移动）。
- **剧情 story**：阵营主线（推进解锁新区域，与世界编年史联动）。

## 6. 生命周期规则（用户明确分三类）

新增单位生命周期分级 `unit.LifecycleClass`：

```go
type LifecycleClass string
const (
    LifecycleProtagonist LifecycleClass = "protagonist" // 玩家控制角色——永不衰老死亡（主角恒在，命运可玩性根基）
    LifecycleFunctional  LifecycleClass = "functional"  // 固定功能性NPC(商人/传送管理员/任务发布/铁匠)——永不衰老死亡(世界服务恒定)
    LifecycleMortal      LifecycleClass = "mortal"      // 普通NPC(路人/村民/野外散人)——会衰老死亡(世界新陈代谢)
)
```

### 6.1 规则
- **玩家角色（protagonist）**：`Age` 可增长（叙事用，她会成长/成熟）但**永不触发自然死亡**。战斗死亡走既有 `LivesRemaining` 机制（命运层另有救济），衰老不夺命。
- **功能性 NPC（functional）**：商人、传送管理员、任务发布者、城镇铁匠等**职能固定的 NPC**——`Age` 冻结或增长但**永不死亡**，保证世界服务（买卖/传送/接任务）恒定可用。落 `faction_spawn.go` 的据点 NPC 中按职能标记。
- **普通 NPC（mortal）**：村民、路人、野外散人——`Age` 随世界 tick 增长，到高龄按确定性掷骰自然死亡（补上之前审计的"衰老→年龄增长→自然死亡"缺口），死亡触发血脉传承/世界编年史记录。**世界因此有新陈代谢**：老的逝去、新的出生（接既有 pregnancy/newborn 路径）。

### 6.2 实现
- `faction_spawn.go` 落 NPC 时按职能赋 `LifecycleClass`。
- 衰老结算（新 `aging.go` 或扩展 `personality_drift.go`）：部署边界对 `mortal` 单位 `Age++`，高龄（如 >70 + 确定性掷骰）触发 `ReasonCharacterDied`（已有 reason code），经 Mutator/lives 路径，best-effort 入世界编年史。
- `protagonist`/`functional` 在衰老结算里**显式跳过死亡判定**（statuslint 友好：年龄非受保护字段；死亡走既有 lives 白名单路径）。

## 7. 世界编年史（全新，与角色编年史并列）

**现状**：`chronicle.go` 是**单位级**——记录某角色的人生大事（已有，`GET /api/sessions/:id/units/:unitId/chronicle`）。

**新建**：**世界级编年史**——记录整个世界的纪元大事，独立于任何单个角色。

### 7.1 数据
```go
// 新表 world_chronicle（或复用 worldbus 存储扩展）：world_id + world_tick + 事件
type WorldChronicleEntry struct {
    ID         string
    WorldID    string
    WorldTick  int
    Era        string  // 纪元（按重大事件分段，如「开拓纪元」「三阵营之战」）
    Category   string  // faction_war/boss_slain/town_fall/hero_born/hero_died/zone_unlocked/cataclysm
    TitleZH    string  // 「秩序之军攻陷晨曦城」
    NarrativeZH string // 一段史官口吻的叙事（规则文案 + 可选 LLM 润色，有 fallback）
    ActorRefs  []string// 涉及的角色/阵营/区域 id
}
```

### 7.2 入史规则（世界事件自动记录，代码驱动）
- 区域 boss/世界 boss 被讨平 → `boss_slain`。
- 阵营冲突/城镇易主 → `faction_war`/`town_fall`。
- 传奇角色诞生/陨落（高 importance）→ `hero_born`/`hero_died`（接生命周期与血脉传承）。
- 新区域解锁/阵营主线推进 → `zone_unlocked`。
- 纪元更替：累计重大事件达阈值，开启新纪元（叙事分段）。
- 全部经 `events.EmitProcessEvent` 旁路 + 写 world_chronicle，**不改保护字段**。

### 7.3 展示
- 命运客户端新增"世界编年史"入口（司命台亦可看）：按纪元倒序浏览世界大事。
- 与角色编年史并列：角色史=她的人生，世界史=她所处时代的洪流。

## 8. 导航与传送（前端）

### 8.1 三层地图
- **世界地图**：缩略看全部区域（区域块 + 阵营色 + 等级带 + 她当前所在区高亮），点区域 → 若可达则前往（传送/边界）。
- **区域地图**：当前区域全屏 hex（现有 FateBoard，按 `currentZoneMap` 渲染）。
- **小地图（minimap）**：左上角，当前区域她周边缩略 + 城镇/boss/任务标记，点击移动主视角。

### 8.2 性能前提（视野裁剪）
区域地图可比现在大（如 40×30）。前端 `bootstrapPixi` 必须**只渲染视口内的格子**（viewport culling）——否则大图卡顿。这是大地图的硬前提，随阶段1 一起做。

### 8.3 传送
- 城镇传送门：她到过的主城/城镇间可互相传送（`ZonePortal kind=portal`，解锁制——到过才能传）。
- 区域边界：走到边界格 → 过渡到相邻区域（`kind=border`）。
- 传送/过渡 = 切 `CurrentZoneID` + 落点坐标，端点 `POST /api/sessions/:id/units/:unitId/travel {to_zone_id, to_coord}`，校验可达/解锁。

## 11. 技术不变量与风险

**不变量（必守）**：
- 受保护字段（HP/Hunger/Morale/Loyalty/LivesRemaining/Mood）恒经 `status.Mutator` + reason code 登记。
- 确定性随机一律 FNV（世界生成、怪物等级、任务投放、衰老掷骰、传送），禁全局 rand。
- 决策用 LLM、结算用代码；新能力 flag 渐进、默认关需可运行时开（避免"flag 默认关=玩家永远玩不到"的历史教训）。
- 玩家直驱写入口走 `guardPlayerAction` 五道门（直驱锁/执行互斥/终局/归属本局/归属玩家方）。

**风险**：
- **多区域改造面大**：`state.Map` 调用点遍布渲染/移动/采集/POI/威胁——靠 `currentZoneMap` 投影 + 渐进替换降风险，旧单图存档零影响兜底。
- **前端性能**：大区域图必须视野裁剪，否则卡——阶段1 硬前提。
- **任务进度结算**：用世界事件回调驱动（击杀/采集/到达时递增），不引入轮询，避免性能与一致性问题。
- **世界编年史体量**：按 world_tick 增长，需容量上限 + 重要度保留（仿角色记忆衰减），避免无限膨胀。

## 12. 待用户确认的设计取舍

1. **区域数量与布局**：默认 1 中立新手区 + 三阵营各 1 主城区 + 若干野外区（共 6-8）。是否够，还是要更多区域？
2. **等级上限**：默认 Lv1-30。MMORPG 体感够吗？
3. **玩家角色衰老**：永不死亡已定；`Age` 是否仍随时间增长（叙事用，她会变成熟）还是完全冻结？
4. **任务是否带阵营主线**：除了支线讨伐/收集，是否要三阵营各一条主线剧情（推进解锁区域 + 写入世界编年史）？
