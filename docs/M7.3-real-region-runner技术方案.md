## 0. 一句话目标

让角色在「不打仗的时候」也持续**自主活着**：一个按真实时钟低频推进的世界 tick 调度器，按冷热分层唤醒单位做日常决策（谋生/社交/移动），**绝大多数走纯代码反射层、极少数关键节点才上 LLM**（§9：真实 LLM 调用 <2%），高代价不可逆动作不自动执行而进命运收件箱等玩家。**默认关闭、灰度开启、零行为回退**。

## 1. 核心架构决策（⚠️ 需用户拍板的取舍）

### 决策 A：region-runner 与现有战棋执行循环的关系 —— **推荐「环境层 + 让位战斗」**

现状：单位决策的**唯一**驱动是战斗执行循环（`session.resolveExecution`，deployment↔execution 相位机 + ATB），由 HTTP `AdvancePhase` 触发、异步 goroutine 跑（`launchAsyncExecution`/`resolveExecutionAsync`，进程级注册表 `isAsyncExecutionRunning` 防重入）。大世界要的是「角色**在战斗之外**也持续生活」。

| 方案 | 描述 | 取舍 |
|---|---|---|
| **A1（推荐）环境层 + 让位** | region-runner 是**新增的环境生活层**：只驱动「**不在战斗中**」的单位做日常决策；一旦某会话进入聚焦战斗（`isAsyncExecutionRunning(sessionID)` 或单位 `in_combat`），region-runner **跳过该会话的单位**，把它们交还现有战斗循环。两层**互不抢同一单位**。 | ✅ 加性、低风险、可灰度；现有战斗逻辑零改动。MVP（§10）正是此形态。**缺点**：两套决策入口并存（战斗用 `generateUnitDecision` 战斗上下文，环境用新的日常上下文）。 |
| A2 统一调度 | region-runner 成为顶层驱动，战斗变成它调度下的一种「聚焦模式」，吃掉相位机。 | ❌ 大重写、行为变更面广、回滚难。违背 §8.6「复用优先、绝不重写」铁律。**不建议现在做**。 |

> **建议采用 A1**。下文均按 A1 展开。A2 留作远期（战斗系统本身要重构时再议）。

### 决策 B：worker 池的进程模型 —— **推荐「单调度 goroutine + 共享 worker 池」**

- **1 个调度 goroutine**（region-runner）：按真实时钟 tick，遍历活跃 region，推世界 tick、拉到点单位、入队决策作业、重排唤醒。**单 region 单写者**（§8.6 风险册：SQLite 用区锁跑 MVP）——MVP 单 region==单 session，天然串行。
- **1 个共享 worker 池**（K 个 goroutine，K≤`scheduler.DefaultMaxInFlight`=64）：各自循环 `agentqueue.ClaimNextJob`（原子认领，已实现）→ 跑决策 → 应用 → `CompleteJob` → 重排该单位下次唤醒。**全局在途上限**由 `scheduler.AdmitDecision` + `CountJobsByStatus(running)` 把守。
- 解耦理由：「**决定谁醒**」（per-region 串行、便宜）与「**跑决策**」（全局并发、可能慢/调 LLM）分离，吞吐与正确性都更好。

### 决策 C：MVP 节奏与范围 —— **推荐「单 region==单 session，慢 WARM 节奏，先只做日常 L1+L3」**

- 1 世界 tick = 真实 **N 秒**（建议默认 N=30~60s，可配 `QUNXIANG_REGION_TICK_SECONDS`），先只跑 WARM 节奏（§10 MVP）。
- 决策范围先做「动机栈 L1（生理/安全：吃/睡/避险）+ L3（社交/移动）」，**不做** L2 高频 reassessGoal（降本）。
- 高代价不可逆动作（叛变/卖传家宝/突然恋爱/送死）**一律不自动执行**，走冻结清单 → 命运收件箱（复用已落地的 `fate.go` `SurfaceFateEvent`/待决策档）。

## 2. 系统架构与数据流

```
                          ┌─────────────────────────────────────────────┐
   真实时钟 ticker ──────▶│  RegionRunner（单 goroutine，单 region 单写者）│
   (每 N 秒)              │  for each 活跃 region:                        │
                          │   1. world.AdvanceTick(region.worldID) → tick │
                          │   2. agentqueue.ListDueWakes(region, tick)    │
                          │   3. 对每个到点单位（跳过战斗中/已死）:        │
                          │      - RemoveWake(unit)          ← 防重复入队  │
                          │      - 若 AdmitDecision(running<64):           │
                          │          EnqueueJob(unit, tick)               │
                          │        否则：背压，下 tick 再来                │
                          └───────────────┬─────────────────────────────┘
                                          │ agent_decision_jobs (pending)
                          ┌───────────────▼─────────────────────────────┐
   共享 worker 池 ───────▶│  Worker × K（K≤64，各自循环）:                │
                          │   job = ClaimNextJob() (原子 pending→running) │
                          │   tier = scheduler.ClassifyTier(...)          │
                          │   decision = decide(unit, tier):              │
                          │     - L1 护栏/反射层 decision.Router → 零 LLM │
                          │     - 仅 HOT 关键节点 → generateUnitDecision  │
                          │       (LLM, 经 budget gate + 归因校验)        │
                          │   applyAmbientDecision(decision)  ← 经 Mutator│
                          │     高代价动作 → 冻结 → fate 收件箱（不执行） │
                          │   TouchLastActiveTick(unit, tick)             │
                          │   tier',next = scheduler.PlanNextWake(tick,…) │
                          │   EnqueueWake(unit, next, tier')  ← 重排       │
                          │   CompleteJob(job, done/failed)               │
                          └───────────────────────────────────────────────┘
```

**单位唤醒一次的完整生命周期（关键：杜绝重复作业）**：
`在唤醒队列(wake_at=t)` → 到点被 runner 拉出 → **先 RemoveWake**（此后不会再被拉到）→ 入队 1 条 job → worker 认领→跑→应用→`TouchLastActiveTick` → `PlanNextWake` 算出 next → **EnqueueWake(next)** 重新入队。一个单位在「被拉出到重排」之间没有 wake 行，故不会被重复入队；job 表里它至多一条 running。

## 3. 决策分层：<2% LLM 的落地（§9 核心卖点）

`decide(unit, tier)` 的三道闸（已有原语，这里组合）：

1. **L1 安全护栏 / 反射层优先**（`engine/decision.Router.Route`，已存在、已影子接入 `reflex_shadow.go`）：日常绝大多数情形（吃饱了待命、按既定目标移动、无威胁）由纯代码反射层直接出意图，**零 LLM**。把现在的影子模式（只测量 `ReflexStats`）在 region-runner 路径**扶正为真短路**。
2. **冷热分层降级**（`scheduler.ClassifyTier`）：WARM/COLD 单位即便遇到本该决策的节点，也走规则 fallback（`fallback*`/`confusedUnitDecision`），不上 LLM；只有 **HOT**（刚卷入事件/战后/关系剧变）单位的**关键节点**才升级 LLM。
3. **预算闸**（复用 `llm_budget.go` 的预算护栏 + 后续 `ai.Service.BudgetGate`）：账户累计成本触顶后，该 region 后续决策全部降级 fallback。

> 已有的归因校验（`SetAttributionEnforcement`/`prepareAttribution`）在 LLM 路径继续生效：环境层突然恋爱/叛变必须有可解析前因，否则优雅回退安全决策。这正好与「高代价动作进收件箱」叠加成双保险。

## 4. 接入前置项落地（agentqueue 包头登记的 3 项 DoD）

| 前置项 | 落地方案 |
|---|---|
| **① 保留期/隐私清理纳入两表** | `PurgeExpiredSessionData` / `EraseSessionPrivateData`（`privacy.go`）按 session 清理时，MVP（region==session）按 `region_id = sessionID` 加 `DELETE FROM agent_wake_queue / agent_decision_jobs`；并在单位死亡路径（`WorldizeDeath`/`recordWorldizedKill` 附近）调 `agentqueue.RemoveWake(unitID)` + 标记其 job 失效。**接入第一步就补**，否则留孤儿违保留期。 |
| **② stale-running 回收** | 新增 `agentqueue.ReclaimStaleJobs(ctx, db, olderThan, maxAttempt)`：把 `status=running 且 claimed_at < now-olderThan` 的作业 `attempt+1` 后置回 `pending`（超 `maxAttempt` 置 `failed`）。region-runner 每 M 个 tick 调一次。防 worker 崩溃把 job 永卡 running、抬高背压致饿死。需在 `claimed_at` 上加索引（接入时迁移）。 |
| **③ 多世界 region 作用域** | `ListDueWakes` 增 `world_id` 维度（签名 `ListDueWakes(worldID, regionID, ...)`，SQL 加 `AND world_id = ?`），避免跨世界同名 region 串唤醒。MVP region_id 全局唯一（==sessionID）故当前无碰撞，但**接入时一并改**以免落地后再回溯。 |

## 5. 与现有系统的协调与安全

- **让位战斗（决策 A1 的执行）**：runner 拉到到点单位后，**逐一过滤**：`isAsyncExecutionRunning(sessionID)`（该会话在聚焦执行）或 `unit.Status.InCombat` 或 `life_state != active` → 跳过（不入队、保留/重排其 wake 到稍后），把单位让给战斗循环 / 不打扰已死者。
- **单写者并发**：MVP 单 region 单写者天然无竞争。worker 池对**作业**的并发已由 `ClaimNextJob` 原子认领保证（已有 8-goroutine race 测试）；对**同一单位状态**的并发由「一单位至多一条 running job」+ 状态变更全程经 `status.Mutator`（行级读改写）兜底。
- **优雅退出**：region-runner + worker 池由 `main.go` 启动（新增 goroutine），监听现有的 `signal.NotifyContext` ctx；shutdown 时 ctx 取消 → runner 停止派发、worker 跑完手头 job 即退（在途 job 留 running，下次启动由 stale-reclaim 回收）。
- **panic 恢复**：worker 循环每个 job 包 `recover()`（仿 `resolveExecutionAsync` 的 panic 恢复 + 45min 超时），单个 job 崩溃不拖垮 worker；崩溃 job 经 stale-reclaim 重试。
- **确定性**：唤醒/分层/节奏全确定（`engine/scheduler` 纯函数）；决策内的随机仍用 `sessionID+tick+actor` 哈希（项目既有确定性随机约定），保证可复算。

## 6. 分步实施计划（每步可独立提交 + 评审，shadow→灰度→开启）

> 纪律同 M6/M7：每步绿（build/-race/statuslint/vet）+ 多智能体对抗评审 + 补前置项。

- **M7.3-real-0｜前置项补齐（纯加性，先做）**：`ReclaimStaleJobs` + `claimed_at` 索引迁移；`ListDueWakes` 加 `world_id`；`privacy.go` 两条清理路径纳入两表 + 单位死亡 `RemoveWake`。**不接循环**，纯能力 + 测试。
- **M7.3-real-1｜RegionRunner + worker 池骨架（默认关闭）**：`internal/regionrunner` 新包（goroutine 生命周期、ctx 取消、real-clock ticker 注入式时钟便于测试）；`main.go` 在 flag `QUNXIANG_REGION_RUNNER_ENABLED` 为真时启动。**空转**：推 tick、拉到点、入队、worker 认领后**只记日志不应用**（影子），观测 `/healthz` 新增的 runner 遥测（tick 数、入队/认领/在途、跳过原因分布）。
- **M7.3-real-2｜环境决策应用（灰度）**：worker 真应用日常决策——先只接 L1 反射层（零 LLM、低风险动作：移动/待命/采集），经 Mutator 留痕（用 M7.2 新 reason-code），`TouchLastActiveTick` + 重排。高代价动作走冻结→收件箱。仍 flag 灰度。
- **M7.3-real-3｜HOT 关键节点上 LLM**：HOT 单位关键节点接 `generateUnitDecision`（环境上下文），过 budget gate + 归因校验；观测真实 LLM 占比是否 <2%。
- **M7.3-real-4｜seed 入队 + 全链路**：单位创建/接入世界时 `SetUnitScope` + 初始 `EnqueueWake`；跑通「建世界→接入→低频自主生活→玩家上线看高光卡/待决策」。
- **（并行）M7.3-real-5｜切 MySQL**：dbdialect 已就绪，砍 SQLite 单连接瓶颈（业务零改，§8.6 阶段 2 后半）。

## 7. 可测试性策略（背景实时系统怎么测）

- **注入式时钟**：RegionRunner 不直接调 `time.Now`/`time.Ticker`，而是接受一个 `Clock`/tick 信号 channel；测试用手动 tick 驱动，断言「一个 tick → 拉到点 → 入队 → worker 处理 → 重排」确定性发生，无需真等真实时钟。
- **纯逻辑已全测**：`engine/scheduler` 已有 6 测试；`agentqueue` 已有原子认领/并发 race 测试。
- **集成测试**：SQLite 起临时库，手动推 N tick，断言队列状态流转、背压触发、stale-reclaim、让位战斗（mark 一个 session async-running 后其单位被跳过）。
- **遥测断言**：`/healthz` runner 块（tick/enqueued/claimed/inflight/skipped_by_reason/llm_rate）作为灰度观测面，影子期先看数再开应用。

## 8. 风险登记（映射 §9 风险册）

| 风险 | 缓解 |
|---|---|
| 无限 agent LLM 成本爆炸 | 三道闸叠加（反射层真短路 + 冷热降级 + budget gate）+ worker 全局在途 64 + 背压；影子期先量 LLM 占比再开。 |
| 离线角色「灾难性」后果致挫败 | 冻结清单：高代价不可逆动作不自动执行，进命运收件箱等玩家；归因校验双保险；死亡仅走衰老/被动卷入。 |
| region-runner 与战斗循环抢单位 | A1 让位：`isAsyncExecutionRunning`/`in_combat`/`life_state` 三重过滤跳过；两层互不写同一在战单位。 |
| worker 崩溃/卡死 | per-job recover + 超时；stale-running reclaim（attempt+上限）；优雅退出在途留 running 下次回收。 |
| 拆表/双写不一致 | 前置项先行；单 region 单写者；新增全程经 Mutator/agentqueue 原子原语；影子期观测对账。 |
| SQLite 单连接吞吐瓶颈 | MVP 单 region 单写者 + 小 worker 池可跑；阶段 2 后半切 MySQL（业务零改）。 |
| 默认开启误伤现网 | **flag 默认关闭**；shadow（只记不应用）→ 灰度（只 L1）→ 全量；每步可回退。 |

## 9. 不做（明确划界，避免范围蔓延）

- 不做 events 三层归档 / 算力账户分 tier 路由 / 编年史按需渲染——阶段 3。
