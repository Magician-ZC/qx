## 1. 地图点击回归——根因确诊（运行时实测，对抗复核通过）

**现象**：点世界地图格子不再弹「地块信息 + 住户」面板（333b89f 引入的功能）。

**根因**（playwright 实测 100% 确定性复现）：不是某次提交改坏了链路，而是潜伏交互 bug 被「全屏可拖相机」武装后必然发作：

- `bootstrapPixi.ts:289-292` `handleTileClick` 是所有点击总闸：`if (didDrag) return;`。
- 全文件**唯一**把 `didDrag` 复位为 false 的地方是 stage 的 `pointerdown`（:344-349）；`endDrag` 只清 `dragging` 不清 `didDrag`。
- 单位 token 绑定在 `pointerdown` 且先 `event.stopPropagation()`（:1086-1092）→ stage 的 pointerdown（唯一复位点）被短路。
- 于是：**拖一次图（didDrag=true）之后，点任何「人」永远没反应**，且连续点人无法自愈（每次都 stopPropagation），只有先在空格按一下才复位——吸收态。
- 主世界 24×16 图在 zoom=1.3 下宽约 1971px > 视口，她和全部住户默认在屏外左上角出生聚落 → **看人必先拖图** → 必然触发。

**叠加因素**（同样要修）：
1. didDrag 阈值过敏：累计曼哈顿 |dx|+|dy|>5px 即吞点击，触控板普通点击被概率性误杀。
2. 首屏相机不以她为中心：新玩家先看到一张空地图，必须拖图找人。
3. 六边形 hitArea 角缝死区（verticalStep 2% 压缩所致），实测部分坐标点不中任何格。
4. 运维：用户正在跑的 :8080 后端进程早于 a2942d7（map-pois/move 路由 404）——**验收前必须重启后端到 HEAD**。

**修复方案**（W1，文件：`frontend/src/game/bootstrapPixi.ts`）：
- 主修：token 绑定从 `pointerdown+stopPropagation` 改为与地形格一致的 `pointertap`、**不再 stopPropagation**（:1089-1092）。按 token 时 pointerdown 正常冒泡复位 didDrag，吸收态消失，且可从 token 起手拖图。
- **禁忌**：不得改成在 pointerup/endDrag 里清 didDrag——pointertap 在 pointerup 之后派发，那样拖拽误触防护会完全失效。
- 加固1：阈值 5px → 9px 且改欧氏距离。
- 加固2：首拍快照到达时把 boardOffset 初始化为「以她为中心」（screenCenter − herTileCenter×scale），仅 spectator 模式且仅首次。
- 加固3（可选）：对话 CTA hitbox（:1787-1793）的 pointerdown 也去掉 stopPropagation（保留其 pointertap 的语义）。
- 回归验证：拖图→点她 token 应弹「就是她」面板；拖图→点 NPC 应弹住户面板；真拖拽中不应弹；战棋（非 spectator）视图点单位选中仍正常。

## 3. 地块事件系统技术方案（A2~A5、A9、A13、A14）

**总思路**：不造新引擎。三套已通的结算链（production 建造/采集、threat+engine/encounter PvE、random_events 分支模板）用 `player_actions.go` 的「玩家直驱」范式接到地块上。决策层=玩家点击即玩家意志（**零 LLM**），结算层全走既有 `execute*/Resolve*`（受保护字段恒经 Mutator），叙事层复用 LIFE_BEAT + fate feed（WS `fate_life_beat` 前端已接）。

### 3.1 后端新文件 `backend/internal/session/tile_affordances.go`

`Service.TileAffordances(ctx, sessionID, unitID, q, r) (TileAffordances, error)`：
- 复用 `terrainSupportsActivity/terrainSupportsStructure`（production.go:667-700）做动作白名单——**与 LLM 候选同口径，不另写一套规则**。
- 输出：地形/地标/POI(含 consumed)/本格 Structure(进度/收获就绪)/占据者/与 unit 的六边形距离/动作清单。
- 动作清单元素：`{action, activity?, structure_type?, target_unit_id?, label_zh, available, reason_zh}`——不可用也返回（置灰显示原因：「她还没走到那里」「材料不够」「已采掘殆尽」）。
- 动作族：`gather`(farm/fish/forage/hunt/mine 按地形)、`build`(5 设施按地形+材料)、`harvest`(本格农田就绪)、`forge/upgrade`(本格己方完工铁匠铺)、`demolish`、`poi_encounter`(本格未消耗 POI)、`talk`(同格/相邻具名 NPC)、`trade`(同格/相邻 NPC)。
- 大多数动作要求 `unit_on_tile`（站上去才能做）；talk/trade 允许相邻。

### 3.2 后端新文件 `backend/internal/session/tile_actions.go`

`Service.ExecuteTileAction(ctx, sessionID, unitID, req) (TileActionResult, error)`：
- 校验：unit 归属本会话（同 player_actions 范式）、`ExecutionInProgress` 互斥（409「她正忙于眼前的事」）、actor 站在目标格（不在则「她还没走到那里」）、affordance 可用性复检（防直接 POST 绕过）。
- 结算：直接构造决策载荷调既有 `executeGather/executeBuild/executeForge/executeUpgrade/executeDemolish`（饥饿经 Mutator、打猎风险 HP 经 ReasonCombatHit、背包发放、appendLog 全在里面）；harvest 走农田收获既有路径。
- 资源 POI 加成：`gatherRewards` 处读同格未消耗资源 POI（与 map_pois.go 的 poiRoll 同口径判定），命中则奖励 ×1.5 并标 consumed。
- 落痕：appendLog + `events.EmitProcessEvent`（流程事件，kind=player_tile_action）+ WS 推送。
- Save 持久化 + 返回结算摘要（中文一句话 + 资源增减明细）。

### 3.3 后端新文件 `backend/internal/session/poi_encounter.go`

`Service.ResolvePOIEncounter(ctx, sessionID, unitID, q, r) (POIEncounterResult, error)`：
- 用 `npcEventTypeFor` 同口径解析该格 POI 事件类型，按类型路由：
  - **埋伏 ambush** → `ResolveEliteEncounter`（threat.go:531，多回合消耗战/分赃/惩罚/收件箱全通）——这就是命运客户端的 PvE 地图入口（A14）。
  - **行商 merchant** → 返回行商货单（复用 trade.go 定价），前端弹交易面板走 A13 的 trade 端点。
  - **求助 help / 奇遇 adventure** → 借 `randomEventBranch` 模板结构（random_events.go:32-46，含 Wallet/Hunger/Morale/物品增量）做 2~3 分支确定性裁决（FNV(sessionID+coord+turn+salt)），LLM 仅叙事（失败走规则文案 fallback）。
  - **迷途 lost** → 结识：建立初始关系（relation 四轴小幅正向，经既有 relation 路径）+ 叙事。
- 状态变更经 `applyStatusMutation` 同路径；新 reason code 在 `reason_codes.go` 登记：`POI_ENCOUNTER_RESOLVED`（流程）、状态变更尽量复用 `ECONOMY_LOOT/EMOTION_REWARD/COMBAT_HIT`。
- 结算后该 POI 标 consumed（幂等防重放闸）。

### 3.4 POI 消耗态（改 `map_pois.go` + `types.go`）

- `State` 加 `ConsumedPOIs map[string]int`（"q,r" → 消耗时 turn；JSON 整块随 session 存储，无需迁移）。
- `MapPOIs` 下发每个 POI 附 `consumed: bool`。
- 前端 `drawPOIs` 对 consumed 徽标 alpha 0.35。

### 3.5 `player_actions.go` 改造（A5/A6/A7/A8/A10/A11）

- `PlayerMoveUnit`：
  - 加 `ExecutionInProgress` 互斥（A7，所有玩家直驱入口统一加）。
  - 限距：单次六边形距离 ≤ 8 格（A6，超出报「太远了，先让她走近些」）。
  - 成功后 best-effort 调 `surfaceEncounterBeatBestEffort`（A5）+ appendLog 落痕（A8）。
- `PlayerEquipItem`：互斥 + 落痕。
- 新增 `PlayerUseItem`（A10）：消耗品结算复用既有吃食物路径（饥饿恢复经 Mutator + ReasonSurvival 系 code），非消耗品报错。
- 新增 `PlayerUnequipItem`（A11）：槽位→背包 + 重算派生攻防（镜像 `EquipBackpackItem`，放 `internal/unit` 同文件）。

### 3.6 HTTP 路由（`router.go`，与既有单位作用域路由同段集中注册）

```
GET  /api/sessions/:id/tile-affordances?unit_id=&q=&r=     → TileAffordances
POST /api/sessions/:id/units/:unitId/tile-action            → {action,q,r,activity?,structure_type?}
POST /api/sessions/:id/units/:unitId/poi-encounter          → {q,r}
POST /api/sessions/:id/units/:unitId/use-item               → {item_id}
POST /api/sessions/:id/units/:unitId/unequip                → {slot}
POST /api/sessions/:id/units/:unitId/trade                  → {target_unit_id,mode:buy|sell,item_id,quantity}
```
（q/r 走 query/body 而非 path，避免路由歧义；错误一律中文友好提示，同 move 范式。）
**注意**：router.go 是热点文件，路由注册由主循环统一接线，实现 agent 不直接动它。

### 3.7 前端接线

- `session/api.ts`（热点，单 agent 独占）：新增 `getTileAffordances/executeTileAction/resolvePOIEncounter/useItem/unequipItem/tradeWithUnit` 六个客户端函数 + 类型。

- `game/bootstrapPixi.ts`（单 agent 独占）：W1 回归修复全部项 + consumed POI 降透明度。

### 3.8 硬约束（全体实现 agent 必守）

1. 受保护字段（HP/Hunger/Morale/Loyalty/LivesRemaining/Mood）**恒经 status.Mutator**；新 reason code 先在 `reason_codes.go` 登记。改完必须 `go run ./cmd/statuslint ./...` 零退出。
2. 确定性随机一律 FNV(sessionID+coord+turn+salt)，禁全局 rand。
3. 注释一律中文、文件头 `// 文件说明：…`；标识符英文。
4. 玩家直驱零 LLM（叙事可选用 LLM 但必须有规则 fallback，不得因 LLM 不可用阻断结算）。
5. **不挂默认关的 feature flag**——本批功能是玩家直驱、零 LLM 成本、确定性结算，无灰度风险；审计已证明「flag 默认关=玩家永远玩不到」是本项目最大历史教训。
6. 前端 fate.css 不改（FateBoard 内联样式贴墨色调）；styles.css 不改。
