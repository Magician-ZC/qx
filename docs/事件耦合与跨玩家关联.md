# 《群像》事件耦合与跨玩家关联

> 回答两个问题：**(A) 世界的事件如何与角色关联**（深化设计宪法 [`设计宪法-角色命运开盒.md`](设计宪法-角色命运开盒.md) §4.1 的 M1 翻译层）；**(B) 不同用户的角色如何靠事件相互关联**（共享世界的全新维度）。全部建立在现有 `events`/`relations`/`region`/`engine/arbitration`/`engine/decision` 之上。与宪法冲突处以宪法为准；本文是宪法 M1 的可实现化 + 跨玩家层的补全。

---

## 0. 核心思想

**一个角色对世界保有一组「锚」**——她在乎的人、她的红线、她的目标、她的债/仇/爱、她所在的地方。**世界事件只是探针；命运 = 探针点亮一根锚的那一刻。**

- 这把「世界为角色命运服务」从口号变成机制：事件来时只在该角色的锚上做 **O(命中)** 匹配，而非对全世界算分；未命中 = 世界自己消化、玩家永不可见（无限事件→几十件的物理实现）。
- 耦合是**双向**的：我的角色的行为也被「世界化」成探针，去点亮别人的锚——「我背叛 B → B 的仇恨锚被点亮 → B 据此自治反应 → 又成为对我的新探针」，闭环全靠 `events` 唯一事实源。
- 跨玩家关联的第一性问题不是「怎么撮合」，而是 **「先造一层凌驾于 session 之上的共享世界总线」**——因为现在 `events`/`relations`/`units` 全被 `session_id` 沙盒化，两个玩家的角色**物理上不在同一事实源里**。

## 1. 世界事件 ↔ 角色（问题 A，深化 M1）

### 1.1 相关性锚索引 `relevance_anchors`（把「对她有意义」预先物化）
为每个玩家角色维护一张常驻锚表，事件来时 O(命中) 而非 O(全世界)：

`relevance_anchors(player_unit_id, anchor_kind, anchor_ref, weight[0..1], decay_half_life_days, source_event_id, created_turn, updated_turn)`

`anchor_kind` 六类，与宪法 Relevance 五因子 + attribution 六 cause 对齐：

| kind | ref | weight 来源 | half-life |
|---|---|---|---|
| `relation` | 对方 unitID | `relations` 聚合 closeness/hostility 归一 | ~14d |
| `redline` | 离线宪章红线 ID | 固定高 | 永不衰减 |
| `goal` | L2 目标 ID | 目标强度 | 跟随目标 |
| `debt_grudge_love` | 债/仇/爱清单项 ID（来自 §4.5 village_seed hooks） | 强度 | 慢 |
| `geo` | 当前 RegionID | 1.0 | ~3d |
| `legacy` | pinned 物 / 血脉项 | 高 | 永久 |

锚不是静态：每次 `status.Mutator` 落一条改变关系/目标/债务的事件，就 upsert 对应锚的 weight 并刷新 `updated_turn`（复用 events 已有的 Actors/ReasonCode）。

### 1.2 相关性评分 + 事件→命运 beat 翻译
每根命中锚的贡献 `contribution = clamp01(anchor.weight · relativeImportance(kind) · time_decay · hop_fidelity)`，相关性用 **noisy-OR** 组合：`Relevance(E,P) = 1 − Π_命中锚 (1 − contribution)`。其中 `relativeImportance` 由宪法因子相对大小 `{熟识0.32, 红线0.28, 目标0.18, 债仇爱0.14, 地理0.08}` 归一化（最重要类别=1.0），`time_decay = 2^(−Δdays/half_life)`，`hop_fidelity` 见 1.4。**≥0.35 进候选**。
> ⚠️ **修正宪法 §4.1 的标定缺陷**：宪法原写「加权求和」，但因权重和为 1，**单根锚的贡献被自身权重封顶**（最强的关系锚也只到 0.32 < 0.35），导致「密友倒地」单凭关系锚永远进不了前台。已改为「相对重要度归一化 + noisy-OR」：单根强锚即可过阈、多锚共振次可加、相关性恒在 [0,1]、对任一 weight 单调。已落地 `engine/relevance` 并有蒙特卡洛分布/单调测试守护。

每个 `(reason_code × anchor_kind)` 配一条 data-driven `translation_templates`，把系统事实改写成对她的命运 beat：
- `COMBAT_DOWN × relation`（命运绑定密友倒地）→ 「老吴在北岭前线倒下了，生死未卜」，`force_pending=true`。
- `RELATION_BETRAYAL × relation`（密友被第三方背叛）→ 「{friend} 那边出事了」+ worry 压力位 + 生成 `debt_grudge_love` 锚。
- `ECONOMY_LOOT × geo`（她所在 region 被劫）→ 「商路被断，她攒的本钱要打水漂」+ 改 goal 锚可达性。

模板产物即高光卡/待决策的 `narrative_zh`，复用宪法 §5.4 渲染管线。

### 1.3 双向世界化 `Worldize`（我的角色行为 → 影响他人的世界事件）
`status.Mutator` 落事件后挂钩子：当 `actor` 是玩家角色且 `reason_code ∈ worldizing_codes` 白名单（背叛/救援/劫掠/倒地/社交/债务），就**反查「谁的锚会被这件事点亮」**——对 target/旁观者跑一次反向 1.1，命中即给他们落一条 `PROPAGATION_INBOUND` 探针，并据此驱动其 NPC 自治（`decision.Router` 的 `StrategicFork/SocialOffer` 置真）。入向与出向**用同一张传播规则表**，不是两套逻辑；玩家的影响力**没有特权**（付费不增传播半径/强度）。

### 1.4 关系图传播与衰减失真 `Propagate`（沿 relations 图 N 跳）
事件命中某人的锚后，沿 `relations` 图向「关心他的人」BFS 传播：边权 = closeness 归一（敌视边也传播——「我仇人的仇人出事了」也是命运）。每跳 `hop_fidelity ×= 0.6`（可信度衰减），EmotionalTag 随 salience 失真 → 渲染「据说/传闻」（对齐宪法不可靠叙事）。**停跳三条件**：`hop≥2`（最多 2 跳，防全图雪崩）/ `transmit<0.15` / `fidelity<0.3`。每跳落 `PROPAGATION` 痕迹（hop/fidelity/source_event_id），使「越传越失真」成为**可仲裁数据**而非编造。二手消息因 `hop_fidelity` 已乘进 Relevance，天然更难进前台——「密友倒地=直击，远房表亲的邻居出事=被自动过滤的背景噪声」。

### 1.5 NPC「有意义事件」预算（锚加权调度，从源头不产噪音）
后台 NPC 事件生成器若均匀掷事件 → 90% 是噪音。改为 **anchor-weighted budgeting**：每 tick 的世界事件预算按「该 region/该 NPC 的玩家锚密度」加权——一个 NPC 被越多玩家角色当作锚指向，越可能被掷中重大事件。村子半径 1 region（§4.5 的 20 人网）是锚密度天然最高区，世界事件自然聚焦在「她在乎的人和地方」。配额上限：单角色每日入向探针 ≤12（超出按 Relevance 截断丢弃，世界仍在 events 里发生，只是不打扰她）。
> **破圈预算（防信息茧房）**：每日强制 ≥1 件来自零锚来源（陌生人/新 region）的低权事件有资格进高光卡（geo + 新鲜度加权），作为新锚的种子——这同时是问题 (B) 跨玩家相遇的撮合入口。

### 1.6 喂给 FateScore 与命运收件箱
`Relevance(E,P)` 直接充当 `FateScore = 不可逆度 × 牵挂相关度 × 情绪强度` 里的**牵挂相关度**项；三档路由（<0.25 自治 / 0.25–0.55 高光卡 / ≥0.55 待决策）、待决策 ≤3/日、过期兜底回响卡全继承宪法。新增 `RELEVANCE_MATCH/INBOX_ENQUEUED` 留痕，使「为什么这件事进了我的收件箱」可追溯到具体锚命中（点开看「因为他是你的密友(熟识0.71)+这事断了你的商路(目标0.4)」）。

## 2. 跨用户角色的事件关联（问题 B，新维度）

### 2.1 World Bus 共享世界总线（打破 session 沙盒的前提）
新增 `worlds` 表 + `units` 增列 `world_id`（同 world 才可能相遇）。**跨玩家事实写到 world 级的 `cross_events` 表，不写进任一方 session 的 `events`**；各自 session 的 M1 订阅 `cross_events`，翻译成本地的高光卡/待决策（echo）。

> **不可篡改硬不变量**：`cross_events` 是跨玩家**唯一事实源**（append-only，带权威 `occurred_at` = 谁先动手、`prev_cross_event_id` = 复仇/证据链反指针）。双方 session 内只存它的**翻译产物 echo**，**永不互相直接 UPDATE 对方的 units/relations**——一个玩家无法篡改另一个玩家的角色。任何跨玩家写入只能产出 cross_event，各自的 `status.Mutator` **只改自己 owner 一侧**的 relations。用 storage 层断言 + code review 守死。

### 2.2 社会客体撮合 `social_objects`（密度优先，不随机配对）
`social_objects(type, region_id, member_unit_ids, severity, expires_at)`，type ∈ `{common_enemy, disaster, debt_chain, marriage_pact, contested_seat, blood_feud, trade_route, alliance}`。

撮合不是「2000 人里随机选两人」，而是：**后台世界先自然产出社会客体**（灾难落在 region / 一个 NPC 债主把 A、B 都列进债务链 / 共同仇敌 NPC 升级），再从「与该客体地理近 + 关系网有交集」的候选里**确定性择人**：

`MatchScore = 0.35·地理近 + 0.30·钩子契合(LifeGoal/债仇爱命中 type) + 0.20·关系网交集 + 0.15·密度调节(活跃跨玩家关系越少越高，防大R垄断社交)`

`≥0.45` 入池 → `arbitration.Resolve(Key=worldID+SO.id+epoch, Score=MatchScore)` 取前 N 名（**确定性、可复现、与谁先在线无关**，反「付费更高频先抢社交位」）。撮合冷却：每角色每日新绑定跨玩家社会客体 ≤2。**撮不到时由后台 NPC 社会客体兜底，不强求玩家相遇。**

### 2.3 七种交互统一协议
结识/结盟/交易/联姻/反目/复仇/开战 共用一条管线：发起方 session 产出 `CrossIntent(initiator, target, type, terms, region, tick)` → 经 `consent_gate` 分级 → 落一条 `cross_event` → 双方各自的 `status.Mutator` 在自己 session 内按 `cross_relation_templates` 做**本侧** relations 增量：

| 交互 | 关系增量（data-driven，仅改本侧） |
|---|---|
| 结识 | trust+1/affection+1 |
| 结盟 | trust+3 + 建 alliance 社会客体 |
| 交易 | arbitration 定价；成功 trust+2 / 违约 trust-4·rivalry+3 |
| 联姻 | affection+4/trust+3 + 血脉绑定 hook（影响祖魂传承） |
| 反目 | trust-3/rivalry+4 |
| 复仇 | 触发方 rivalry+5；被复仇方进收件箱 |
| 开战 | faction 级 rivalry+fear；落到个人 = 「她的盟友/家人被卷入」 |

### 2.4 异步同意三档 `consent_gate`（知情权，复用后果分级闸）
B 对**离线**的 A 的角色发起交互时，按「该交互对 A 的最坏后果落在后果分级闸哪一层」决定流程：

| 后果层 | consent 档 | 行为 |
|---|---|---|
| 层1 可恢复（结识/小额交易/反目降温） | **UNILATERAL** | 直接成立，事后进 A 收件箱（NOTIFY，祖魂只「知情+一瞥」） |
| 层2 高代价（结盟/大额债务/复仇宣告/偷袭背叛） | **UNILATERAL_BUT_CONTESTED** | 成立但 A 上线得一张「待决策」回应卡（还手/求和/认账，后果对称） |
| 层3 不可逆且双向（联姻/血脉绑定/生死契） | **REQUIRES_CONSENT** | A 同意前只挂 pending；A 上线由**她的角色按归因自治决定**是否接受（不是玩家点「同意」按钮，祖魂只能劝） |

超时（A 长期不上线）：REQUIRES_CONSENT 类自动失效 + 给 B 回响卡「她的命没有等到回应」；CONTESTED 类按 A 的离线宪章兜底自治回应。

### 2.5 祖魂自洽（被影响 = 她的命遇到另一个人的命）
**B 永远只能写一条 cross_event（改变 A 的处境），改不了 A 的 units/relations/灵魂。** A 上线后，收件箱里 B 造成的事件统一用祖魂语气渲染——不是「玩家 X 背叛了你」，而是「她在桥头被她信任的人推了一把——是另一缕命撞上了她的命」。A 的角色把这条 cross_event 当作一个新的**现实压力位/关系恶化记忆**，经现有 `decision`+`attribution` 管线**自治产出回应**（归因必须落到 relation 恶化或 pressure，否则 OOC 回退），祖魂只能在「待决策」里给 2 选项（还手/隐忍）。彻底贴合宪法 D1：**B 改变的是处境，A 的角色仍有自己的命、自己的怕、自己的选。**

### 2.6 跨玩家零和裁决（异步公平，反 P2W）
两个玩家角色争夺排他结果（同一 NPC 的联姻对象/同一势力的继承席位/同一批战利品）→ `arbitration.Contest(Key=worldID+SO.id+tick, Score=各自投入)`。**Score 严格只由角色投入构成**（相关属性 + 与标的的 relations + 地理近 + 已投入的 cross_event 数沉没投入），**付费档不进 Score**。Resolve 给确定性胜负 + 全序排名（败者拿「退而求其次」的社会客体补偿）。胜负在裁决 tick 统一结算（非先到先得）；离线方有离线宪章自动投入兜底 + 裁决前最短补投窗口。蒙特卡洛红线：付费方 5×频率+插队跑 1e4 次，胜率须 ≈Score 比例 ±2%。

### 2.7 共享历史与可仲裁证据链
一条 cross_event 还沿两侧关系图传播（复用 1.4）：B 偷袭 A → A 的密友 C（可以是第三个玩家的角色）按 relations 强度收到衰减后的 `CROSS_DERIVED`「你信任的人被人偷袭了」→ 撮合系统据此生成 `blood_feud` 社会客体，把 A、C 绑成对抗 B 的同盟。**三方 session 里的 echo 都指向同一 `cross_event_id`**——这就是「三个玩家都记得的那次背叛」，且 `occurred_at` 证明 B 先动手。echo 是视角化叙事层（可罗生门），但**事实唯一**：争议回退到 `cross_events` 原表仲裁，「谁先动手」恒由 `occurred_at` 判定。

## 4. 与现有设计/引擎的咬合
- **M1/FateScore**：本文是 §4.1 的可实现化——`Relevance` 直接充当「牵挂相关度」，三档路由/每日预算/回响卡全继承，不另造分。跨玩家事件与后台 NPC 事件走**同一条策展管线**，玩家分不出对方是 NPC 还是另一个玩家（保持「前台是一个角色的命运」）。
- **arbitration**：撮合择人与跨玩家零和都直接调 `arbitration.Resolve`，频率无关/反 P2W 原样继承。锚/传播只决定「谁会收到这件事」，**不决定胜负**。
- **attribution**：Worldize 触发的 NPC 反应、A 对跨玩家事件的回应，都照常过 `ValidateAttribution`——跨玩家因复用 `relation`/`pressure` 两类 cause，无需新归因类别。
- **后果分级闸**：`consent_gate` 三档映射 `consequence_gate` 三层——**别的玩家也杀不死你 D0 的新角色**，跨玩家不可逆（联姻/血脉）同样要 A 侧牵挂≥70 且在世≥7 天才解锁。
- **祖魂 / 留痕**：见 §2.5、§2.7。

## 5. 风险登记（合并两 facet）
| 风险 | 缓解 |
|---|---|
| 跨玩家直接写对方 session → 篡改别人的角色 | 硬不变量：跨玩家只产 append-only cross_event，各自 Mutator 只改本侧；storage 断言禁 cross-session UPDATE |
| 撮合撮不到 or 洪泛（大R垄断社交） | 密度优先 + 社会客体钩子驱动；MatchScore 含密度调节项；每日新绑定 ≤2；撮不到由 NPC 社会客体兜底 |
| 异步交互被滥用骚扰离线玩家 | consent_gate 层2/3 门槛 + 同一对玩家每日高代价交互 ≤N 冷却；后果分级闸硬锁不可逆；祖魂叙事重写「被针对」为「两缕命相撞」；角色可自治搬离 region 解除地理近 |
| 零和异步不公平（在线/付费方抢先做高 Score） | arbitration Key 含 tick 频率无关、裁决 tick 统一结算；离线宪章自动投入兜底；Score 不含付费/插队 |
| 传播洪泛/回声室震荡（报复-示好无限循环） | ≤2 跳硬截断 + 0.6^hop 衰减 + 衍生事件仍过 Relevance≥0.35；Worldize 对同一 (actor,target,code) 设 1 日冷却 + 每跳 importance 单调衰减 |
| 信息茧房（世界对她坍缩成 20 人小圈） | 破圈预算：每日 ≥1 件零锚来源低权事件可进高光卡，作新锚种子 |
| 共享历史罗生门（两玩家记的不是同一件事） | echo 仅视角叙事层，强制带 cross_event_id + occurred_at；争议回退唯一事实源仲裁 |
| 付费撬动相关性 = 变相 P2W | 传播半径/锚密度调度/相关性阈值/每日探针预算全列**不可售**（与 arbitration 同级红线）；蒙特卡洛证伪付费方关键事件到达率与免费档无显著差 ±2%；可售仅叙事密度（更长的「为什么是我」追溯、更美的传播演出） |
| 翻译模板覆盖不全 → 命中锚却无 beat | `translation_templates` 按 (reason_code×anchor_kind) 建全覆盖矩阵；缺模板回退 `DefaultReasonText` 保守 beat + 计入遥测排期补；force_pending 类必须有专属模板 |

## 6. 已落地：relevance 引擎原语
本文最「结算」的一块（纯确定性数学）已实现为 `backend/internal/engine/relevance`（纯逻辑、可测试，延续「决策用 LLM、结算用代码」）：
- `Score(hits, hopFidelity)`：锚命中 → 相关性，用 **relative-importance 归一化 + noisy-OR**（`1−Π(1−contribution)`）组合——**修正了宪法 §4.1「加权和为 1 致单锚封顶过不了阈」的标定缺陷**：单根强锚即可过阈、多锚次可加、恒在 [0,1]、对 weight 单调；`PassesGate`（≥0.35）。
- `RelativeImportance` / `TimeDecay` / `HopFidelity(hop)=0.6^hop` / `StopPropagation`（≤2 跳、transmit/fidelity 地板）。
- `ConsentTierFor(worstLayer)`：后果层 1/2/3 → unilateral/contested/requires_consent。
- `MatchScore(...)`：撮合四因子加权。
共 10 个测试（含蒙特卡洛分布/单调性）：`go test ./internal/engine/relevance`。它与 `arbitration`（裁决）、`decision`（归因/反射）一起，构成结算层确定性的几块基石。

## 7. 落地阶段建议
1. **先做单人 M1（问题 A）**：relevance_anchors + 评分 + translation_templates + Worldize（对 NPC）+ FateScore 接入。这在**当前 session 沙盒内即可跑**，不需要 World Bus，直接让「世界事件落到我的角色命运」成立。
2. **再做 World Bus（问题 B）**：worlds + world_id + cross_events + 不可篡改不变量，这是跨玩家的地基（对应大世界设计 §8 的作用域升级）。
3. **然后社会客体撮合 + 七种交互 + consent_gate + 跨玩家 arbitration**。
4. **最后跨玩家传播 + 共享历史证据链**。
> 关键：问题 A 能在不动 session 架构的前提下先落地见效；问题 B 必须等 World Bus，与大世界的 region 化演进同步。
